# Java基础面试题答案

## 1. Java 有哪些核心特点？
Java 的核心特点通常包括跨平台、面向对象、自动内存管理、生态完善和安全性较好。跨平台依赖 JVM，面向对象让代码更易维护和复用，自动垃圾回收降低了手动管理内存的成本。

## 2. Java 为什么能够实现跨平台？
Java 源码会先被编译成字节码文件，字节码不直接依赖具体操作系统。不同平台只要安装了对应实现的 JVM，就可以把同一份字节码翻译成该平台可执行的机器码，因此实现“一次编写，到处运行”。

## 3. JVM、JRE、JDK 分别是什么？它们之间有什么关系？
`JVM` 是 Java 虚拟机，负责执行字节码。
`JRE` 是 Java 运行环境，包含 JVM 和运行 Java 程序所需的核心类库。
`JDK` 是 Java 开发工具包，包含 JRE 以及编译器、调试器等开发工具。
关系可以理解为：`JDK` 包含 `JRE`，`JRE` 包含 `JVM`。

## 4. Java 为什么既可以说是编译型语言，又可以说是解释型语言？
Java 源码先通过 `javac` 编译成字节码，这体现了编译型语言的特点。字节码在 JVM 中执行时，既可能由解释器逐条解释执行，也可能由 JIT 即时编译成本地机器码执行，所以它同时具有解释型和编译型的特征。

## 5. Java 中参数传递为什么只有值传递？
因为 Java 方法调用时传递的永远是“值的副本”。如果传的是基本类型，复制的是具体数值；如果传的是对象，复制的是对象引用的副本。修改副本指向不会影响原引用，但通过副本操作同一对象的内部状态会影响原对象。

## 6. Java 中基本数据类型有哪些？分别占多少字节？
`byte` 1 字节，`short` 2 字节，`int` 4 字节，`long` 8 字节，`float` 4 字节，`double` 8 字节，`char` 2 字节，`boolean` 官方规范没有明确规定存储大小，逻辑上表示真和假。

## 7. 自动类型转换和强制类型转换有什么区别？
自动类型转换是从小范围到大范围、从低精度到高精度的安全转换，比如 `int` 到 `long`。强制类型转换是从大范围到小范围的转换，需要显式声明，可能发生精度丢失或数据溢出。

## 8. 为什么金融场景中通常不建议使用 `float` 或 `double` 做精确计算？
因为二进制浮点数无法精确表示很多十进制小数，比如 `0.1`。这会导致计算结果出现精度误差，金融场景对精度要求高，通常使用 `BigDecimal` 来避免这类问题。

## 9. `BigDecimal` 为什么推荐使用字符串构造？
因为如果使用 `double` 构造，误差会先随着 `double` 的二进制表示被带入 `BigDecimal`。使用字符串构造可以直接按十进制精确创建对象，结果更可靠。

## 10. 什么是自动装箱和自动拆箱？它们会带来什么问题？
自动装箱是基本类型自动转换为包装类，例如 `int` 转 `Integer`；自动拆箱是包装类自动转换为基本类型。它会带来额外对象创建、性能损耗，以及在拆箱 `null` 时可能出现空指针异常。

## 11. 为什么 Java 中需要 `Integer` 这样的包装类？
因为泛型和集合只能处理对象，不能直接处理基本类型；同时包装类还提供了很多实用方法，比如 `parseInt()`。包装类也让基本类型能够参与面向对象编程。

## 12. `int` 和 `Integer` 有哪些区别？
`int` 是基本类型，性能更高，默认值是 `0`。`Integer` 是引用类型，可以为 `null`，可以用于泛型、集合和需要对象的方法参数中，但会带来装箱拆箱和空指针风险。

## 13. 什么是 `Integer` 缓存？范围是多少？
Java 对 `Integer` 的部分小整数做了缓存，默认范围是 `-128` 到 `127`。在这个范围内通过 `Integer.valueOf()` 创建对象时，会复用缓存对象而不是新建对象。

## 14. 面向对象的三大特性是什么？
封装、继承、多态。

## 15. 封装的意义是什么？
封装是把属性和方法组合在一起，并对外隐藏内部实现细节。它可以提高安全性、降低耦合度，并让对象的使用方式更清晰。

## 16. 继承有什么优点和限制？
优点是代码复用、层次清晰、便于扩展。限制是 Java 只支持单继承，一个类只能直接继承一个父类，但可以实现多个接口。

## 17. 多态是什么？它的实际价值是什么？
多态指同一个父类引用或接口引用，运行时可以表现出不同子类对象的行为。它的价值在于提高扩展性和解耦能力，让代码更符合开闭原则。

## 18. 方法重载和方法重写有什么区别？
重载发生在同一个类中，方法名相同但参数列表不同，是编译期多态。重写发生在父子类之间，子类重新实现父类方法，是运行时多态。

## 19. 向上转型和向下转型分别是什么？向下转型有什么风险？
向上转型是子类对象赋值给父类引用，通常是安全的。向下转型是把父类引用转换成具体子类类型，必须保证对象真实类型匹配，否则会抛出 `ClassCastException`。

## 20. 抽象类和普通类有什么区别？
抽象类不能直接实例化，可以包含抽象方法和普通方法，主要用于定义模板和公共能力。普通类可以直接创建对象，必须给出完整实现。

## 21. 接口和抽象类应该如何选择？
如果强调行为规范、弱耦合、支持多实现，优先用接口。如果强调代码复用、共享状态或模板能力，适合用抽象类。很多场景是接口定义能力，抽象类复用通用实现。

## 22. `String`、`StringBuilder`、`StringBuffer` 有什么区别？
`String` 不可变，适合少量字符串操作和常量场景。`StringBuilder` 可变但线程不安全，性能高，适合单线程频繁拼接。`StringBuffer` 可变且线程安全，但性能通常低于 `StringBuilder`。

## 23. 为什么 `String` 适合作为 `HashMap` 的 key？
因为 `String` 不可变，创建后内容不会变化，因此其 `hashCode()` 结果稳定。作为 key 时不会因为内容变化导致哈希定位失效，这非常适合哈希表使用。

## 24. `HashMap` 的底层数据结构是什么？
在 JDK 1.8 中，`HashMap` 底层是数组 + 链表 + 红黑树。元素先存入数组桶位，发生哈希冲突时形成链表，链表过长且容量达到条件时会树化为红黑树。

## 25. `HashMap` 的 `put` 流程大致是怎样的？
先根据 key 计算 hash，再定位数组下标。如果该位置为空，直接插入；如果不为空，就比较 key 是否相同，相同则覆盖旧值；不同则在链表或红黑树中继续查找并插入。插入后如果元素数量超过阈值，会触发扩容。

## 26. `HashMap` 的 `get` 流程大致是怎样的？
先根据 key 计算 hash 并定位桶位。如果桶位为空直接返回 `null`；如果首节点匹配则直接返回；如果是树节点就按红黑树方式查找，否则遍历链表，通过 `hashCode()` 和 `equals()` 找到目标节点并返回 value。

## 27. `HashMap` 为什么要求 key 正确重写 `equals()` 和 `hashCode()`？
因为 `HashMap` 先通过 `hashCode()` 定位桶，再通过 `equals()` 判断是否为同一个 key。如果只重写一个、不重写另一个，可能导致查找失败、重复 key 无法识别，或者数据覆盖异常。

## 28. `HashMap` 的 key 可以为 `null` 吗？value 可以为 `null` 吗？
可以。`HashMap` 允许一个 `null` key，也允许多个 `null` value。之所以 `null` key 只能有一个，是因为多个相同 key 会相互覆盖。

## 29. `HashMap` 为什么线程不安全？
因为它的 `put`、扩容、链表或树结构调整等操作都没有同步控制。多线程同时操作时，可能出现数据覆盖、元素丢失、结构异常等问题。JDK 1.7 中扩容时甚至可能形成环形链表。

## 30. `HashMap` 的扩容机制是怎样的？默认负载因子为什么是 `0.75`？
当元素个数超过 `容量 * 负载因子` 时会触发扩容，通常扩容为原来的 2 倍，并重新分配元素位置。`0.75` 是时间和空间上的折中值，既能保持较低冲突率，也不会太浪费内存。

## 31. `HashMap` 的容量为什么通常是 2 的幂？
因为这样可以通过 `(n - 1) & hash` 快速计算下标，效率高于取模运算。同时在扩容时，也更方便根据某一位是否为 1 来判断元素是否需要移动到新位置，提高 rehash 效率。

## 32. `HashMap` 和 `Hashtable` 有什么区别？
`HashMap` 线程不安全、效率高，允许一个 `null` key 和多个 `null` value。`Hashtable` 线程安全，但大部分方法都加了 `synchronized`，性能较低，而且不允许 `null` key 和 `null` value。

## 33. `ConcurrentHashMap` 是如何保证线程安全的？
它通过更细粒度的并发控制来保证线程安全。JDK 1.8 中主要结合 `volatile`、CAS 和 `synchronized` 实现，在尽量减少锁竞争的同时保证并发更新安全。

## 34. JDK 1.7 和 JDK 1.8 的 `ConcurrentHashMap` 实现有什么区别？
JDK 1.7 采用分段锁 `Segment` 的设计，相当于把大 Map 拆成多个小段分别加锁。JDK 1.8 取消了 `Segment`，改为数组 + 链表/红黑树，并结合 CAS 和 `synchronized` 实现更细粒度的并发控制。

## 35. 为什么 `ConcurrentHashMap` 中既使用了 `synchronized`，也使用了 CAS？
CAS 适合竞争不激烈、可快速完成的无锁更新，开销小。`synchronized` 更适合处理哈希冲突、链表或树节点修改这类复杂临界区。两者结合能够在性能和正确性之间取得平衡。

## 36. Set 集合为什么可以保证元素不重复？
因为 Set 底层会基于 `hashCode()` 和 `equals()` 判断元素是否已存在。插入时如果发现相同元素已经存在，就不会重复保存，所以能保证元素唯一性。
## 37. JVM 的类加载过程包括哪些阶段？其中“加载、链接、初始化”分别做了什么？
类的生命周期通常包括：加载、链接、初始化、使用和卸载，其中链接又分为验证、准备、解析三个阶段。

加载阶段主要是通过类的全限定名获取该类的二进制字节流，将其读入内存，并在方法区中生成类的元数据信息，同时在堆中生成对应的 `Class` 对象，作为访问该类元数据的入口。

链接阶段分为三步。验证是检查字节码是否符合 JVM 规范，保证类的正确性和安全性；准备是为类变量也就是 `static` 变量分配内存并设置默认初始值，这里通常是零值而不是程序员写的显式赋值；解析是把常量池中的符号引用替换为直接引用，也就是把名字引用转换成真正的内存地址或偏移量。

初始化阶段才真正执行类构造器 `<clinit>` 方法，对类变量进行程序员定义的赋值，并执行静态代码块。它不是调用对象的构造函数；对象构造函数 `<init>` 属于对象实例化过程，而不是类加载初始化阶段。
## 38. 什么是双亲委派机制？它解决了什么问题？有没有场景会主动打破它？
双亲委派机制指的是类加载器在收到类加载请求后，并不会先自己去加载，而是先把请求逐级委托给父加载器。只有当父加载器无法完成加载时，子加载器才会尝试自己加载。

它主要解决两个问题。第一，避免同一个类被多个类加载器重复加载，保证类的唯一性；第二，保证 Java 核心类库的安全性，防止用户自定义同名类替换核心类，比如 `java.lang.String` 这类核心类会优先由启动类加载器加载。

需要注意的是，这里并不是“父加载器加载过 String 之后，应用类加载器就发现 String 已经被加载了”，而是应用类加载器一开始就会把加载请求委派给父加载器，所以核心类根本轮不到它自己去加载。

有些场景会主动打破双亲委派。比较典型的是 Tomcat，为了让不同 Web 应用加载各自独立的类，通常会为每个应用使用独立的类加载器，实现类隔离。另一个典型场景是 JDBC SPI。像 `DriverManager` 这种由启动类加载器加载的类，需要去加载第三方数据库驱动，而驱动类通常在应用类路径中，所以会借助线程上下文类加载器去加载，这本质上也是一种对传统双亲委派模型的突破。
## 39. Java 运行时数据区域有哪些？堆、虚拟机栈、方法区分别存什么？
Java 运行时数据区域通常包括：程序计数器、Java 虚拟机栈、本地方法栈、堆、方法区，另外在 HotSpot 中还经常会提到直接内存，但它严格来说不属于 JVM 运行时数据区的一部分。

堆是线程共享的，主要用来存放对象实例和数组，是垃圾回收的主要区域。

Java 虚拟机栈是线程私有的，随着线程创建和销毁。它由一个个栈帧组成，每次方法调用都会创建一个栈帧，里面通常包含局部变量表、操作数栈、动态链接、方法返回地址等信息，所以它保存的是方法调用过程中的运行数据。

方法区是线程共享的，主要存储类的元数据信息，比如类结构信息、运行时常量池、字段信息、方法信息、静态变量、JIT 编译后的代码等。不同 JDK 版本对方法区的具体实现不同，例如 HotSpot 早期是永久代，JDK 8 以后更常提元空间。
## 40. 可达性分析为什么比引用计数法更适合 Java 的垃圾回收？
引用计数法会为对象维护一个引用计数，每当有一个地方引用它时计数加一，引用失效时计数减一；当计数为零时就认为对象可以回收。它实现思路直观，但无法很好地解决循环引用问题。比如两个对象互相引用，即使它们已经不再被外部使用，引用计数也不会变成零，因此无法被回收。

可达性分析则是从一组称为 `GC Roots` 的根对象出发，沿着引用关系向下搜索。如果一个对象能够从 `GC Roots` 出发被访问到，就说明它仍然是可达的，不能回收；如果无法被访问到，就说明它已经不可达，可以被回收。常见的 `GC Roots` 包括虚拟机栈中引用的对象、方法区中类静态属性引用的对象、本地方法栈中 JNI 引用的对象等。

相比引用计数法，可达性分析更适合 Java，核心原因是它能够正确处理循环引用问题，因此 Java 主流虚拟机都采用可达性分析来判断对象是否存活。

## 41. JVM垃圾回收中，哪些对象可以作为GC Roots？请详细说明每种GC Roots的具体来源。
在可达性分析算法中，GC Roots 是一组必须存活的对象，作为搜索的起点。主要包括以下几类：
1. **虚拟机栈中引用的对象**：各个线程调用方法时，栈帧中的局部变量表里引用的对象。
2. **方法区中类静态属性引用的对象**：类的静态变量引用的对象。
3. **方法区中常量引用的对象**：字符串常量池中的引用，以及被 `final` 修饰的常量引用的对象。
4. **本地方法栈中 JNI 引用的对象**：Java Native Interface 中引用的对象。
5. **Java 虚拟机内部的引用**：基本数据类型对应的 Class 对象，常驻的异常对象（如 `NullPointerException`、`OutOfMemoryError`）等。
6. **被同步锁持有的对象**：被 `synchronized` 关键字锁住的对象。

这些对象之所以作为 GC Roots，是因为它们要么是程序运行的基础，要么与外部环境有直接关联，必须保证它们存活。

## 42. Spring框架中，Bean的生命周期包括哪些关键阶段？请描述从Bean定义到销毁的完整过程。
Spring Bean 的生命周期主要包括以下关键阶段：
1. **实例化**：通过构造器或工厂方法创建 Bean 实例。
2. **属性填充**：为 Bean 的属性注入值或引用（依赖注入）。
3. **初始化前**：如果 Bean 实现了 `BeanPostProcessor` 接口，会调用 `postProcessBeforeInitialization` 方法。
4. **初始化**：如果 Bean 实现了 `InitializingBean` 接口，会调用 `afterPropertiesSet` 方法；如果配置了 `init-method`，也会调用该方法。
5. **初始化后**：如果 Bean 实现了 `BeanPostProcessor` 接口，会调用 `postProcessAfterInitialization` 方法。
6. **使用**：Bean 准备就绪，可以被应用程序使用。
7. **销毁前**：如果 Bean 实现了 `DisposableBean` 接口，会调用 `destroy` 方法；如果配置了 `destroy-method`，也会调用该方法。

整个过程中，Spring 容器负责管理 Bean 的创建、依赖注入、初始化和销毁，开发者可以通过各种回调接口参与这些阶段。

## 43. MySQL的InnoDB存储引擎中，MVCC（多版本并发控制）是如何实现的？请解释undo log、read view和版本链的关系。
MVCC 通过以下机制实现：
1. **隐藏字段**：每行数据包含 `DB_TRX_ID`（最近修改该行的事务ID）、`DB_ROLL_PTR`（指向 undo log 的回滚指针）、`DB_ROW_ID`（行ID）等隐藏字段。
2. **undo log**：记录数据修改前的版本，用于事务回滚和 MVCC。每次修改都会生成 undo log，形成版本链。
3. **版本链**：通过 `DB_ROLL_PTR` 指针将同一数据行的不同版本连接起来，每个版本对应一个 undo log 记录。
4. **read view**：事务在第一次执行查询时创建，包含：
   - `m_ids`：当前活跃事务ID列表
   - `min_trx_id`：最小活跃事务ID
   - `max_trx_id`：下一个即将分配的事务ID
   - `creator_trx_id`：创建该 read view 的事务ID

**读取规则**：
- 如果 `DB_TRX_ID < min_trx_id`：该版本在 read view 创建前已提交，可见
- 如果 `DB_TRX_ID ≥ max_trx_id`：该版本在 read view 创建后才开始，不可见
- 如果 `min_trx_id ≤ DB_TRX_ID < max_trx_id`：
  - 若 `DB_TRX_ID` 在 `m_ids` 中：事务仍活跃，不可见
  - 否则：事务已提交，可见

如果当前版本不可见，就通过版本链找到上一个版本继续判断，直到找到可见版本或到达链尾。
